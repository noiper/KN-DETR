# Repository Agent Instructions: KNDETR (Edge-Optimized Temporal Object Detection)

## Project Context & Mission
This repository implements **KNDETR**, a hardware-aware, dual-path temporal object detection architecture based on RT-DETR. 
The core scientific objective is maximizing inference speed on edge devices (e.g., NVIDIA Jetson) without sacrificing detection accuracy. This is achieved by dynamically routing video frames between a heavy "Key" path (full Transformer) and an accelerated "Non-Key" path (lightweight temporal fusion).

## Technology Stack & Environment
- **Language**: Python 3.10+
- **Framework**: PyTorch
- **Baseline Model**: RT-DETR (v2)
- **Target Deployment**: Edge GPUs (NVIDIA Jetson)

## Architectural Ground Rules (CRITICAL)
When generating code, modifying model architectures, or writing tests, you MUST adhere to the following KNDETR design principles:

1. **Dual-Path Execution Logic**:
   - **Key Frames**: Execute the full CNN backbone and Hybrid Encoder to generate robust spatial priors.
   - **Non-Key Frames**: Bypass the heavy encoder. Use a lightweight fusion block to combine raw backbone features (specifically $S_5$) with cached Key features, followed by a scalable decoder parameterized by $n_D$.
2. **Object Query Reuse & Matcher Alignment**:
   - Non-Key frames explicitly reuse fully refined object queries from the Key frame's decoder layer.
   - The Non-Key path MUST share the exact Hungarian Matcher indices generated by the Key frame to ensure temporal features are applied to the correct objects.
3. **Teacher-Student Distillation**:
   - During temporal training, the Key path is strictly **frozen** (acting as the Teacher).
   - Knowledge Distillation is performed at the **feature level** (Teacher CCFF outputs vs. Student fusion outputs), NOT just at the final logit level.
4. **Adaptive Routing Threshold ($\tau$)**:
   - Routing decisions are strictly deterministic, based on feature-level differences (e.g., L1 distance) of $S_5$ features between frames. Do NOT suggest secondary neural networks for K/NK routing.

## Dataset & Preprocessing Standards
- **Classes are STRICTLY 0-Indexed**: All custom dataset scripts and model YAML configurations must use 0-indexing for PyTorch loss function compatibility.

## Coding Standards
- **Tensor Documentation**: You must explicitly comment expected tensor shapes (e.g., `[batch, channels, height, width]`) during complex temporal fusion or concatenation steps.
- **No Silent Overhead**: Avoid unnecessary `.clone()`, `.to(device)` transfers in the training loop, or mathematically expensive operations where lightweight alternatives exist. 
- **Configuration over Hardcoding**: Expose all hyperparameters (like $n_D$, $\tau$, and temporal gap $k$) in YAML config files.
